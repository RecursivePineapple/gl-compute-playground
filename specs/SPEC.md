This tool will be used for experimenting with OpenGL compute shaders. It is a UI tool using react and redux, with minimal fluff. It will have a panel on the left side that contains a 'directory' tree. The tree will have a section for buffers, shaders, pipelines, and visualizers. This tool functions by defining buffers, shaders, pipelines, and visualizers. Opening a pipeline and clicking its execute button will process all of its data, after which the user will be able to examine the buffer contents or load buffers into specific visualizers. Visualizers use very simple opengl to render a mesh, and the opengl is defined by a javascript file which may be edited within the program or externally. A project is just a standard directory of files. Each entity maps to one file, and their contents are plain JSON.

A project is opened by clicking on the 'File' option in the app header bar and selecting 'Open Project', which opens a folder selector dialog. Selecting a folder opens it as a project.

Clicking on an entity in the project tree will immediately open it on the right panel. Edits are auto-saved immediately to the matching file.

A buffer is a simple object that is defined by its data type (f32, f64, i32, i64, u32, u64) and its dimensions (x, y, z). All dimensions default to 1. A 1D buffer has y=z=1; a 2D buffer has z=1. It may also have predefined data, in which case it is read-only. The predefined data may be pulled from a binary file, or it may be pulled from a csv file. Loading from clipboard should also be supported.

A shader uses the same syntax as a standard OpenGL compute shader. It is compiled as-is. The editor must compile the shader as the user edits the file (debounced) and report any syntax errors to the user immediately. The monaco editor is a good candidate for this. The bindings and uniforms for the shader will be automatically parsed, but their contents will be specified by the pipeline.

A pipeline is a collection of shader invocations. It is edited by creating shader and buffer nodes in a graph node editor UI, then linking them. Shaders will have a slot for each binding; the direction (read or write) is determined automatically by parsing the readonly/writeonly qualifier on the binding in the shader source — these qualifiers are mandatory. Read slots connect to a buffer's output; write slots connect to a buffer's input. A buffer may be read by multiple shaders, but may only be written by one shader — connecting a second writer to a buffer is an error. Shader nodes will also have three number inputs for the X/Y/Z dispatch dimensions, each defaulted to 1, and one per uniform.

There is an 'execute' button in the bottom right corner of the pipeline editor graph. When a pipeline is executed, GL buffers are created. Any with pre-existing data will be populated automatically. Shaders are dispatched in topological order based on read/write dependencies. A glMemoryBarrier(GL_SHADER_STORAGE_BARRIER_BIT) is issued between each dispatch. Circular dependencies, or two shaders writing to the same buffer, will show an error to the user upon pipeline execution. Once the pipeline finishes, the contents of the GL buffers are copied back to their buffer entities so that the user can examine their contents or visualize them. Read-only (predefined-data) buffers are not overwritten.

Visualizers are eval'd in the main process directly. A visualizer script must return a method with two parameters and no return value. The signature will be `function(tessellator: Tessellator, buffers: {[bufferName: string]: BufferLike})` where `Tessellator` is a class that builds a VBO and `BufferLike` is a tagged union that may be any of the primitive arrays (`Int32Array`, `Uint32Array`, `BigInt64Array`, `BigUint64Array`, `Float32Array`, `Float64Array`). The `buffers` parameter is the contents of every buffer in the project, and the type of the BufferLike must match the data type of the buffer. The tessellator is backed by a VBO that has a position array, UV array, normal array, and color array. A visualizer has four tabs. The first is the script, the second is the vertex shader, the third is the fragment shader, and the fourth is for the render. The vertex and fragment shaders are both very simple by default, but they follow a similar workflow as the compute shaders (use the monaco editor, and show compilation errors as you type). Whenever the script or the shaders change, the render VBO is automatically updated and its program is recompiled. Whenever a pipeline is executed, the VBO is updated as well.
